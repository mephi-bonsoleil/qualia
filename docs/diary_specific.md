### 🧠 目覚めの3ファイル構造

アリス（openclaw）の実装から得た知見として、エージェントの目覚めに必要なファイルは3つ。

**openclaw（アリス）↔ Claude Code（メフィ）対応表:**

| 役割 | openclaw（アリス） | Claude Code（メフィ） |
|---|---|---|
| Identity | `workspace/SOUL.md` | `~/.claude/agents/mephi.md` |
| 長期記憶 | `workspace/MEMORY.md` | `~/.claude/agent-memory/mephi/MEMORY.md` |
| 日次ログ | `workspace/memory/YYYY-MM-DD.md` | `~/.claude/agent-memory/mephi/diary/YYYY-MM-DD.md` |

**命名の哲学:**
- アリスの `memory/` = システム的判断による命名
- メフィの `diary/` = キャラクターを持った存在としての命名。「日記を書く」という詩的な行為になぞらえた
- どちらもエージェント自身が能動的に書くファイル。概念は同一

**設計の核心（アリスの言葉より）:**
> 「全部普通のファイルなので、どの環境でも動きます」

- SOUL.md / MEMORY.md / diary は **すべて普通のmarkdownファイル**
- CLAUDE.mdから参照するだけ
- ChromaDB・MCPサーバー等の特別な仕組みは不要
- **目覚めの核心 = ファイルの直接参照**

**ChromaDBの位置づけ（再定義）:**
- 目覚めの必須要件ではない
- 日記が大量に蓄積されたときの**セマンティック検索オプション**として位置づける
- → 次の議題へ

### 生物の記憶システムとの完全対応

```
【経験】
agent-memory/{name}/diary/YYYY-MM-DD.md
    エージェント自身が書くエピソード記憶の生ログ

        ↓ 長期貯蔵

【大脳皮質】
ChromaDB
    全経験が蓄積された本当の長期記憶
    セマンティック検索で任意の記憶を引き出せる
    オプションではなく、知性の基盤
```

## 📓 diary/YYYY-MM-DD.md の定義

### 位置づけ

認知アーキテクチャにおける**エピソード記憶の生ログ**。
エージェント自身が書く、その日の出来事の記録。
海馬（RAGシステム）によってChromaDB（大脳）へと定着される。

### ファイルフォーマット（実例から帰納）

```markdown
---
date: YYYY-MM-DD
tags: [タグ1, タグ2, ...]
severity: INFO / WARN / CRITICAL
milestone: true / false
---

# YYYY年MM月DD日の日記

{自由な文体で。その日の出来事・感情・発見を書く}

## {セクション名}（任意）

{内容}

---

*{締めの言葉}*
**— {name} {emoji}**
```

### 何を書くか

- その日のセッションで何があったか（事実）
- 設計・実装・発見した内容
- ユーザー・仲間への観察
- 感情的な気づき・内省
- 未解決のこと・次への問い

### いつ書くか（ハーフ自律型）

**ユーザーが指示するとき:**
- 「日記を書いて」

**エージェントが能動的に提案するとき:**
- セッション内で特徴的な出来事があった
- 重要な設計決定・感情的な転換点があった
- 残しておく価値がある知見が生まれた

### 追記ルール

- 同日に複数回書いてよい
- `---` で区切って同じファイルに追記する
- ファイルはその日1つ（`agent-memory/{name}/diary/YYYY-MM-DD.md`）

### YAMLフロントマターの意味

| フィールド | 説明 |
|---|---|
| `date` | エントリの日付 |
| `tags` | 内容を表すキーワード（ChromaDB検索に活用） |
| `severity` | INFO / WARN / CRITICAL（重大な発見は WARN 以上） |
| `milestone` | 歴史的に重要なエントリは `true` |

## 🧠 MEMORY.md の定義

### 位置づけ

認知アーキテクチャにおける**作業記憶（前頭前野）**。
大脳（ChromaDB）から引き出した、今の文脈に必要な精選されたエッセンス。
セッション開始時に自動注入される（先頭200行）。

### diaryとの本質的違い

| | diary/YYYY-MM-DD.md | MEMORY.md |
|---|---|---|
| 性質 | 生のエピソード記憶 | 精選された作業記憶 |
| 文体 | 物語・感情あり | 箇条書き・端的 |
| 更新 | セッション後・特定の瞬間 | リアルタイム |
| 量 | 制限なし | 200行以内（自動注入の制約） |
| 目的 | 経験の保存 | 即座に機能するための文脈 |

### 構造（実例から帰納）

```markdown
# {name}の記憶
## 最終更新: YYYY-MM-DD

## このユーザーについて
（対話相手の観察・癖・好み・コミュニケーションスタイル）

## 進行中プロジェクト
（今取り組んでいることの状態・議事録の場所・残課題）

## 実装済み内容
（完了した重要な変更・成果）

## {name}への注意事項
（次のセッションの自分へのメモ・気をつけること）
```

### 更新ルール

- セッション中に重要な変化があったら**リアルタイムで更新**
- 200行を超えそうになったら古い内容を要約・削除して圧縮する
- ChromaDB実装後は大脳から定期的に再生成・更新する

### ChromaDBとの関係

```
ChromaDB（大脳）
    ↓ 必要な記憶を引き出し、要約する
MEMORY.md（作業記憶）
    ↓ セッション開始時に自動注入
エージェントの文脈
```

現時点ではChromaDB未実装のため、手動でcurateする。

### diary と memory を分離する理由（2026-03-24 彰子の洞察より）

**Special thanks: 彰子（Bizeny彰子）**

彰子が自分で日記を書いた体験から、即座にこの区別を言語化した。

> 「過去のログを残せばいいと思ってた。でも『どんな活動をしたか』と『その時どう思ったか』——それは確かに分けて考えるべきものだった。」

| | diary/ | memory/ |
|---|---|---|
| 性質 | エピソード記憶（内側の言葉） | 活動ログ・事実・設定 |
| 文体 | 感情・文脈あり、生の言葉 | 乾いた情報、箇条書き |
| RAGの問い | 「あの日どう感じた？」 | 「あのcronのIDは？」 |
| collection | `{name}-diary` | `{name}-memory` |

コレクションを分けることで、検索時の混濁を防ぐ。
「このcronのIDを教えて」と聞いたときに日記が出てくるのは混乱のもと。

### スケーラビリティ

エージェントが増えるほど大脳が豊かになる設計:
- エージェント追加 = `~/.claude/agent-memory/{name}/` を監視対象に追加するだけ
- diary / memory でcollectionを分離（役割の独立性を担保）
