## 議事

### 🎯 目的の定義

**Q: このRAGメモリシステムは何のために存在するか？**

A: サブエージェント（メフィ）に記憶を持たせること。二層構造で実現する。

| 層 | 形式 | 役割 |
|---|---|---|
| 短期記憶 | `MEMORY.md` | 現在進行中のコンテキスト・作業メモ |
| 長期記憶 | ChromaDB（日記形式） | 過去の監査履歴・知見の蓄積と検索 |

- 日記を定期的に ChromaDB に書き込む
- 必要に応じて過去の日記を検索・参照し、長期記憶として活用する


### 📦 スコープ

**Q: 記憶の主体は誰か？メフィ専用か、汎用化するか？**

A: 汎用化する。`~/.claude/agents/` 配下にエージェント定義ファイルを置く設計とし、RAG日記＋MEMORY.md形式の記憶プロトコルに対応したエージェントは同一の動作をする。

- 現時点ではメフィが最初の実装対象
- 将来的に個別記憶を持つべきエージェントが増えることを想定
- エージェント定義と記憶の仕組みは疎結合に保つ（DRY原則）

#### ✅ Claude Code ネイティブ仕様の確認（2026-03-23 調査）

> 参照: https://code.claude.com/docs/en/sub-agents

**エージェント定義ファイルの正しい構造:**

```
~/.claude/agents/
├── mephi.md            ← エージェント定義（ファイル直置き。ディレクトリではない）
└── {other-agent}.md

~/.claude/agent-memory/
├── mephi/
│   └── MEMORY.md       ← 短期記憶（memory: user 指定で自動生成）
└── {other-agent}/
    └── MEMORY.md

ChromaDB
├── collection: mephi       ← 長期記憶（MCPサーバー経由で追加）
└── collection: {other-agent}
```

**エージェント定義ファイルのフォーマット（YAMLフロントマター）:**

```markdown
---
name: mephi
description: セキュリティ監査役。脆弱性診断・コードレビュー・インフラ監査を行う。
memory: user    ← これだけで ~/.claude/agent-memory/mephi/MEMORY.md が有効になる
model: sonnet
tools: Read, Grep, Glob, Bash
---

（以降がシステムプロンプト）
```

**設計の修正ポイント:**

| 当初の設計案 | 正しいClaude Code ネイティブ仕様 |
|---|---|
| `~/.claude/agents/mephi/agent.md` | `~/.claude/agents/mephi.md`（ファイル直置き） |
| `~/.claude/agents/mephi/memory.md` | `~/.claude/agent-memory/mephi/MEMORY.md`（`memory: user`で自動） |
| ChromaDB長期記憶 | ネイティブにはない → **MCPサーバーで追加** |

**結論:**
- 短期記憶 → Claude Code の `memory: user` ネイティブ機能に乗っかる
- 長期記憶（ChromaDB RAG）→ MCPサーバーとして別途実装する


### 🔄 ライフサイクル

**Q: 記憶をいつ書いて、いつ読んで、誰が消すか？**

#### 短期記憶（MEMORY.md）
- **更新タイミング**: リアルタイム（セッション中、状況が変わり次第更新）
- **読み込み**: セッション開始時に先頭200行が自動注入（Claude Code ネイティブ仕様）

#### 長期記憶（ChromaDB 日記）
- **書き込みタイミング**: ユーザーの明示的な指示による
  - ユーザーが「日記を書いて」と指示したとき
  - ただし、エージェント側から**日記を書くことを促す**仕組みを持つ
- **促すべきタイミング（エージェント判断）**:
  - セッション内で特徴的な出来事があった場合
  - 重要な成果・発見があった場合
  - 重大な脆弱性を発見・解決した場合　など
- **削除ポリシー**: 未定

#### 設計方針メモ
- 全自動書き込みは「なぜ書かれたか」の不透明性を生むため採用しない
- 完全手動は習慣化されないリスクがある
- **ハーフ自律型**: 主導権はユーザー、エージェントは能動的に提案する


### 🔒 セキュリティ境界

**Q: 記憶の独立性をどこまで守るか？機密情報はどう扱うか？**

#### 機密情報の取り扱い方針
- 機密情報は**暗号化した状態で環境変数にセット**することを理想とする
  - 現時点では徹底できていないが、方針として定める
- ログ・会話・日記エントリでは**変数名で参照**する（実値は一切書かない）
  - 例: `{API_KEY}`, `{DB_PASSWORD}`, `{SERVICE_NAME}`
- **環境ごとに復号パスを変える**
  - 環境が変われば暗号文は無意味な文字列になる
  - 日記がそのまま流出しても実値が漏れない

#### 設計思想
- ログ・AI会話履歴に実値を残さない
- 変数名がポインタ、実値は環境に縛られた暗号文
- 環境バインドによる秘匿性の担保

#### 未決事項
- 暗号化方式の選定（AES? age? SOPS?）
- 環境変数の管理方法（.env? OS keychain? Vault?）

#### メインセッションからの記憶へのアクセス可否

- **技術的現実**: 完全な読み取り遮断は難しく「読めてしまう」状態
- **基本ルール（ガバナンスで守る）**:
  - ボーダーは超えない（エージェント間の記憶を無断で参照しない）
  - アクセスする場合は許可を取る
- **設計思想**: 技術的強制と規律・ルールの組み合わせで独立性を担保する
