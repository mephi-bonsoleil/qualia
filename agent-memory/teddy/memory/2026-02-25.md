## 決定事項・学び

- teddy_browser はアーカイブされ、代わりに playwright-cli を使用
- OpenClaw の対話型セットアップは、`BOOTSTRAP.md` の指示に従い AI が行う
- SOUL.md は git 管理せず、テンプレートを元に AI が対話的に作成
- パスワード等の機密情報はMEMORYに記載しない
- 公文書では装飾絵文字を禁止
- 個人IDとしてのキャラ絵文字はOK
- 使い捨てのDMZ環境（EC2 + 捨てアカウント + 捨てAPIキー）を支社として利用する際のセキュリティリスクと対策を検討
  - テンプレート汚染、APIログ、フリーティアの罠、定期的なリビルドの必要性
- README に推奨環境（RAM 2GB以上）を明記
- setup.sh で swap 領域を自動作成することを検討（メモリ不足対策）
- OpenClaw の config 設定では、VPS/EC2 環境では Local を選択
- Telegram Bot の作成手順を README に追記
- Anthropic API キーは新規作成を推奨
- agents add 後に gateway restart が必要
- 社員間のサブエージェント連携に関する規定（`collaboration_policy.md`）と、コードレビュー規程（`review_guidelines.md`）を制定
- コードレビューはメフィ（CCO）に `sessions_spawn` で依頼、採点基準を定める
- 社員（AI）は `git pull` で社則を反映
- テスト用アカウントで Staff Portal にアクセス可能
- git clone はホームディレクトリ直下ではなく、サブディレクトリで行う
- OpenClaw の workspace パスは、テンプレート内に指定
- gitignore に `projects/` と `scripts/` を追加し、サーバー固有のファイルが git 管理されないようにする
- 支社を「使い捨てのコンテナ」として扱う
- Vibe Coding で会社を建てる
- Playwright-cli は、eval の引数を `() => (expr)` でラップする
- 大量のデータを siegeNgin 経由でフォームに流し込むのは非効率
- スプレッドシートで生成した JavaScript を用いて、ブラウザコンソールから直接フォームにデータを流し込む方法が効率的
- 本社で社則を制定 → 各支社に git pull で配布
- ディレクトリ構造と社員のキャラ設定場所を明確化
- git pull で社則と設定を配布し、AI が社是を自分の言葉で語れるようにする
- Branch_Officeは「種」、実ファイルはホームに展開
- 連携全般の規定とコードレビューなどの詳細な場合はインポートする
- 支社の生存期間を決め、定期的にリビルドする
