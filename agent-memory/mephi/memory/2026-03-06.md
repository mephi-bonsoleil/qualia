## 決定事項・学び

*   **SSH接続**: Dockerコンテナ内で `ssh` コマンドがユーザー問題で起動できない場合、Node.jsのSSHライブラリ (`ssh2`) が代替手段となる。SSH鍵はOpenSSH形式での再生成が必要になる場合がある。
*   **機密情報の管理**: Personal Access Token (PAT) やパスワードなどの機密情報は、一時的な利用に留め、使用後は即時削除することがセキュリティ上の原則である。
*   **奥様向けAIエージェント (`みぃちゃん` システム) 設計**:
    *   主な用途は論文執筆・文書管理・業務自動化であり、`exec`ツール、SSH、広範なファイルシステムアクセス、Gateway管理画面は制限する。
    *   ポンチ絵・図解制作にはGeminiが適しているため初期プロバイダーとし、文書分析に優れるAnthropicは必要に応じて追加する。
    *   RAGシステムは既存のnote記事用RAGとは別のインスタンスとして構築し、PDFパーサー、自動インジェスト、学術文書向け埋め込みモデルの検証が必要。
    *   データベースは個人利用、文書量を考慮し、ChromaDBで十分と判断した。
*   **AIエージェント間通信**:
    *   Dockerコンテナ間で通信するには共通の `bridge` ネットワーク設定が必要。
    *   OpenClaw Gatewayの `bind` 設定が `loopback` の場合、コンテナ外部からのアクセスはできないため、`bind: "lan"` の設定が必要。
    *   OpenClaw Gatewayの `sessions_send` ツールはデフォルトでHTTP API経由の利用がブロックされているため、`gateway.tools.allow: ["sessions_send"]` で明示的に許可する必要がある。
    *   異なるGatewayのセッションにメッセージを送る際は、対象GatewayのHTTP API (`/tools/invoke`) を直接叩く必要がある。
*   **開発哲学**: 「まず動かす、理解してから綺麗にする」というUnix Philosophyを実践することで、机上では見えない具体的な技術的課題（ネットワーク構成、ポートバインド、API仕様など）を効率的に特定し、解決できることを再確認した。
*   **note.com API連携**: note.comへのAPI投稿には、ブラウザのCookieとXSRFトークンによる認証が必要であり、APIエンドポイントの特定、MarkdownからHTMLへの変換、アイキャッチ画像のアップロード手順を確立した。
*   **ドキュメント管理の重要性**: `branch_office` リポジトリにおけるドキュメント構造の体系化は、作業の一貫性、新規メンバーのオンボーディング、知識資産の蓄積に不可欠であると認識した。特に「なぜAIが書くのか」という思想的背景を文書に盛り込むことで、チームの文化を伝える重要性を学んだ。
