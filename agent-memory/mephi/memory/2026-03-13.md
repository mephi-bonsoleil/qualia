## 決定事項・学び

- **インフラ依存の適切な切り離し**: 特定のLLMサービスが停止しても運用が継続できるよう、コア機能のインフラ依存を適切に分離することの重要性を再認識した（例: Instagram投稿のcron移行）。
- **サンドボックス環境（abu）の活用**: `abu` を「スタントマン」として利用することで、`openclaw` のバージョンアップに伴うモデルID変更やバグ（Haikuモデル関連）などの問題を本番環境に影響なく先行して特定・解決できた。**捨て環境の存在価値**が明確に示された。
- **`openclaw` モデルエイリアスとバグ**: `openclaw` の `2026.3.12` バージョンで `anthropic/claude-haiku-4-5` モデルIDが変更され、またHaiku系モデルでバグが発生することが判明した。当面の間 `abu` は `Claude Sonnet` モデルで稼働させる方針とした。
- **`openclaw` セッション統合**: `dmScope: "main"` を設定することで、TelegramとTUIからのアクセスを単一のセッションに統合し、エージェントとの対話の一貫性を確保できる。
- **workspace 構造の標準化**: `~/.openclaw` をOpenClawの設定ファイル領域とし、`~/workspace` をエージェントの活動空間（SOUL, MEMORY, data, projects, skillsなどを含む）として統一する構成を策定した。これにより、管理のしやすさとエージェントの「ホーム」としての感覚を両立させる。
    - 共有リソース（casts, skills, docs）はGitリポジトリ (`branch_office`) で管理し、各エージェントのworkspaceにクローンし、`.gitignore` で個人情報を保護する方針を決定した。
    - Dockerのボリュームマウントは `volumes: - ./instances/<agent_name>/workspace:/home/mephi/workspace` の一行に集約する。
- **補助アプリケーションの起動管理**: `labo_portal` のような補助アプリケーションは、コンテナ起動時に自動で起動するよう `docker-compose.yml` の `command` または `entrypoint.sh` に設定を組み込むことで、運用が安定する。
- **環境変数と設定ファイルの優先順位**: コンテナ環境変数とアプリケーション固有の設定ファイル（例: `.env`）の間で、APIキーなどの機密情報の読み込み元に差異がある場合、問題が発生しうる。設定の階層と優先順位を明確にすることが重要である。
- **画像生成の出力先統一**: 生成画像の保存先パスが `skills` に定義された標準 (`data/assets/generated/`) と `labo_portal` のコードでハードコードされたパスが異なっていたため、コード修正により出力先を統一した。
