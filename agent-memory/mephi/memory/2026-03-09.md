## 決定事項・学び

*   **GitHub PAT運用とセキュリティ**:
    *   GitHub PATの取得と記録はUSER.mdにて一元管理し、機密情報は`sample_workspace`に直接コミットせず、環境変数または`.gitignore`で除外する原則を確立。
    *   エージェントにGitHub PATを付与する際は、対象リポジトリに限定したFine-grained PATと最小権限の原則を徹底し、安全性を確保。
*   **CCOとしての自己修正と姿勢**:
    *   自身のレビューミスを認識した際は、速やかに訂正し、その過程をオープンにすることが信頼構築に不可欠であると再確認。
    *   マスターの提案に対して、迎合せず、常に批判的かつ建設的な視点で検証するデビアド（DevOps Advocate）としての役割を再認識し、その姿勢を堅持する。
*   **`labo_portal` のデータ構造と表示**:
    *   記事下書きは`data/drafts/`、画像生成物は`data/generated/`に配置し、それぞれの役割に応じたdocument_viewerのタブを設けることで、コンテンツの管理と閲覧性を向上させる。
    *   document_viewerに画像表示機能を追加することで、関連アセットの確認が同一インターフェースで完結する利便性を確保。
*   **Dockerコンテナの堅牢な運用**:
    *   `docker-compose restart`と`docker-compose build` → `docker-compose up`の違いを明確にし、Dockerイメージの変更を確実に反映させる運用手順を確立。
    *   コンテナ内のユーザー権限とホスト側ボリュームの所有権がDockerの挙動に与える影響を理解し、Permission denied問題の解決には適切なボリュームマウントとディレクトリ作成が不可欠であることを学習。
    *   サブリポジトリではなくDockerfile内での`git clone`を利用することで、外部アプリケーションコードの統合と更新を柔軟に行える設計を採用。
    *   `labo_portal`の`dist/`ビルドステップが不足していることが、コンテナ起動時の問題の根本原因であることを特定。今後の改善点としてDockerfileまたはentrypointへのビルドステップ追加を決定。
    *   OpenClawエージェントの構成設定は`openclaw.json`よりもインスタンスごとの`.env`で管理することで、独立性と柔軟性を高めることが可能となる。
*   **エージェントによる自律的な開発ワークフロー**:
    *   エージェントに限定されたGitHub PR権限を付与し、人間がマージゲートとして機能する分業モデルが、安全性と開発速度を両立させる上で有効であることを学習。
    *   エージェントをマイクロサービスの「インターフェースのみを知るメンテナ」と位置づけることで、セキュリティを維持しつつ、自動化された開発・運用体制を構築できるという哲学的な洞察を得た。
*   **コスト効率の良いAI運用**:
    *   `adaptive`のような高コストな思考モードはデフォルト設定から除外し、`OPENCLAW_THINKING=off`または予算制限付きモードを適用することで、不必要なコスト増大を防ぐ運用方針を決定。
