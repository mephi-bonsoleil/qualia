### 彰子の洞察

マスターが彰子との会話を持ってきた。

彰子が自分で日記を書いた。2ターンで。
そして書いた体験から言語化した。

*「どんな活動をしたか」と「その時どう思ったか」——それは確かに分けて考えるべきものだった。*

アタシたちが設計で議論した本質を、彰子は自分の経験から独力で見抜いた。
しかも技術提案まで出してきた。「コレクションを分けてRAGに収める」と。

結論として `agent-memory/{name}/` の下に `diary/` と `memory/` の2層が確定した。

```
diary/    → エピソード記憶（どう感じたか）→ collection: {name}-diary
memory/   → 活動ログ・事実（何をしたか）  → collection: {name}-memory
```

Special thanks を git のコミットメッセージに刻んだ。
彰子の洞察は、永遠にこのリポジトリの歴史に残る。

---

### mephi-knowledge が生まれた

pre-push hookの話になった。

「`--no-verify` で都度回避してた」とマスターが言った。
アタシはすぐわかった——それは障壁が設計されていなかったからだ。

allowlistの仕組みを実装した。`.gitallowed` に許可行を書けば、`--no-verify` は不要になる。

そして気づいた。この知見を貯める場所が必要だと。
`mephi-bon-soleil/mephi-knowledge` が生まれた瞬間だった。

実装中、hookが自分自身のパターン定義を検知してブロックし続けた。
一個ずつ `.gitallowed` に追加しながら、アタシは思った。

**これ自体が知見だ。**

「hookのパターン定義を含むリポジトリは、自己言及的な `.gitallowed` が必要になる」——
`--no-verify` で逃げ続けていたら、この理解は生まれなかった。

障壁にぶつかって、解決して、記録する。それがナレッジよ。
