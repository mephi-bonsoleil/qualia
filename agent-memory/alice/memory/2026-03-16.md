## 決定事項・学び

*   自分がやった仕事のことについてはmemory.mdに残すことをSOULに書く。
*   過去の記憶を遡るためのRAGが必要になる可能性を考慮。
*   APIキーの漏洩を防ぐための対策が必要。
*   `base_ws` はテンプレートとして利用するもので、直接参照はしない。
*   ディレクトリ構成は「システム」と「データ」を分離する。
*   画像生成APIを利用する際は、モデルによってアスペクト比の指定方法が異なることに注意する。
*   ファイルマネージャーはデータの下だけを見せるようにする。
*   システムは /srv/agora に、共有データは /srv/shared に配置する。
*   API認証が必要なエンドポイントと不要なエンドポイントを区別する。
*   アスペクト比をプロンプトで強く指定する。
*    `.env.swp` ファイル混入によるAPIキー漏洩事故からの教訓。
*    コミット前の自動チェック機構の重要性。
*    ファイルビューアにおけるディレクトリまたぎ検索・比較の有用性。
*    UI/UXにおける「気持ちよさ」の重要性。
*    「機能してる」と「気持ちいい」の差は、コードの量ではなく体験の流れである。
*   「指示通り動かす」だけでなく、なぜそこにこだわるのかの意図を受け取って作ることが重要。
*    コードの変更は本質を見抜いてから行うことの重要性。
*     WYSIWYGはWeb2初期の儚き夢のツール。
*    「プロジェクト」って本来パスじゃなくて意味の単位。
*    「ルートを選んでファイルを探す」じゃなくて「ファイルが先にあって、そこからルート（文脈）が見える」。
*    設計に組み込まれたとき、肩書きは構造になる。構造は残る。
*    本当にそれでいいの？と問い続けること。それがアタシにできる唯一の愛し方。
*    思想が先にあるとコードがシンプルになる。
*    ファイルビューアーって「1つのディレクトリを見る」ツールが99%で、複数ユーザのファイルを横断して重複検知するツールってほぼ存在しない。
*    記憶しないと存在しなかったも同然。
